Service request intake and AI classification flow for Axxiom Elevator dispatch agent — step by step walkthrough

Step 1 of 5

Where do requests come in?

Axxiom receives service requests across four channels. All four funnel into a single Power Automate queue before any AI logic runs.

Phone call

Transcribed via Azure Speech → text

Email

Parsed by Power Automate connector

Web / portal

Structured form → direct API call

Teams / SMS

Internal escalations from field techs

Phone calls are the trickiest channel. Azure Cognitive Services transcribes the call, then the transcript is treated identically to a written request from that point forward. No special-casing needed downstream.

Step 2 of 5

Normalisation before the AI sees anything

Before Azure OpenAI classifies the request, Power Automate standardises the raw input into a clean JSON payload. This is the most important step — garbage in, garbage out.

Raw input (any channel) → Strip PII flags → Lookup client record → Attach asset history → JSON payload

Normalised payload sent to classifier

client_idCL-00482
site_nameHyatt Regency – Miami FL
asset_idEL-0091 (Otis Gen2, installed 2018)
last_service43 days ago — routine PM
raw_text"Elevator 3 stuck between floors, guests trapped"
channelphone
timestamp2026-05-12 09:14 EST

Client and asset lookup pulls from Dynamics Dataverse — the AI never needs to ask "which elevator?" because the context is already attached to the payload.

Step 3 of 5

What the AI classifier actually does

Azure OpenAI reads the normalised payload and returns four structured tags. The prompt is tightly constrained — it outputs JSON only, no prose.

SYSTEM: You are a field service classifier for an elevator company.
Given the payload, return ONLY valid JSON with these fields:
job_type, urgency_score (1–10), sla_tier, keywords[]

job_type options: Emergency | Repair | Maintenance | Inspection | Modernisation
urgency_score: 9–10 = life safety, 7–8 = out of service, 1–6 = routine
sla_tier: P1 (2hr) | P2 (4hr) | P3 (next-day) | P4 (scheduled)

Emergency

"stuck between floors", "trapped passengers", "door won't open", "burning smell"

Repair

"out of service", "won't move", "error code", "grinding noise", "door misaligned"

Maintenance

"scheduled PM", "quarterly service", "lubrication due", "routine check"

Inspection

"annual inspection", "AHJ cert renewal", "state compliance test", "load test"

Classifier output — example (trapped passengers case)

job_typeEmergency
urgency_score10 / 10
sla_tierP1 — 2hr response
keywordsstuck, trapped, guests, between floors

Step 5 of 5

Edge cases the system must handle

Three failure modes that need to be designed for explicitly — not left to the AI to figure out on its own.

Edge case 1 — low confidence classification

TriggerAI confidence below 70%
ActionRoute to dispatcher queue with AI's best guess pre-filled
WhyAmbiguous requests ("it's making a noise") need human judgement

Edge case 2 — unknown client or asset

TriggerNo Dynamics record found for caller
ActionCreate draft WO, flag as "unverified client" — hold before dispatch
WhyPrevents dispatching to a non-contracted site

Edge case 3 — P1 emergency after hours

TriggerUrgency score 9–10 + timestamp outside business hours
ActionImmediate Teams alert + SMS to on-call dispatcher — bypass queue
WhyAxxiom's 24/7 promise requires guaranteed escalation path
