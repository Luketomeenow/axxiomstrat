Job date + time accuracy P2 · validation script

Date errors corrupt MTBF calculations and PM interval tracking

Common errors

Date: 2099-01-01 (fat finger)
Date: 1900-01-01 (system default)
Date: future date (pre-scheduled, not updated)
Date: 2022-13-05 (invalid month)
Completion time before start time
Duration: 0 minutes
Duration: 480 hours (typo)

Validation rules

Date must be: between asset install_date and today

Duration must be: between 15 mins and 12 hours

Completion must be: after start time

No future dates on closed work orders

Flag: any date = 1900 or 2099 → NULL + review queue

Run a validation script that checks all 6 rules against every service event record. Flag violations → human review queue with the specific rule that failed shown alongside the record.

Entity 3 — Fault codes (extracted from tech notes)

The hardest clean. The highest value.

Right now, fault codes live inside free-text tech notes. They are completely unanalysable. Extracting and standardising them is the single most valuable data cleaning task in the entire project — it's what turns Axxiom's service history into genuine safety intelligence.

The reason fault code cleaning is so valuable: once every fault is tagged with a standard code, you can ask "what is the most common failure mode on Otis Gen2 hydraulic units over 10 years old in coastal markets?" and get a real answer. Right now that question is completely unanswerable from the data. After cleaning, it's a 3-second query.

Fault code extraction from tech notes P2 · AI extraction

The most important single cleaning task · requires GPT-4o + human review on low-confidence matches

Raw tech notes (actual examples)

"door wouldn't close all the way, adjusted door operator and tracks, left working"

"stuck between 4 and 5, reset controller, passengers ok, needs new motor contactor"

"grinding noise on descent, found worn guide shoes, replaced, tested ok"

"no power to cab, traced to MCB, replaced, all good"

Extracted + tagged

fault_codes: ["DOOR-OP-CLOSE"]
components: ["door operator","door track"]
action: "adjusted"
outcome: "resolved"

fault_codes: ["CTL-RESET","MOT-CONTACTOR"]
components: ["controller","motor contactor"]

fault_codes: ["GUIDE-SHOE-WEAR"]
components: ["guide shoes"]
action: "replaced"

fault_codes: ["ELEC-MCB-FAIL"]
components: ["main circuit breaker"]
action: "replaced"

GPT-4o prompt: "Read this technician note. Extract: (1) fault codes from this taxonomy list, (2) components involved, (3) action taken, (4) resolution status. Return as JSON only. If uncertain, set confidence_score below 0.7."

Taxonomy: pre-build a list of ~80 fault codes covering all major elevator systems — door, drive, control, hydraulic, safety, electrical, mechanical. This list is the critical prerequisite.

AI extract
Confidence ≥ 0.85 = auto-commit. Between 0.70–0.84 = flag for tech supervisor review (they can read notes and confirm). Below 0.70 = sent to senior mechanic review queue — only they know what the note meant.

Entity 4 — Compliance & certificate records

The data that keeps Axxiom legally protected

Compliance records are often the most scattered data in any elevator company — spread across email attachments, a shared drive folder, a coordinator's spreadsheet, and (sometimes) Dynamics. The cleaning job here is consolidation and standardisation.

Certificate expiry date P1 · consolidate from all sources

The most operationally critical field · live compliance alert system cannot function without this

Where it currently lives

— Email attachments (PDF certs)
— Coordinator's Excel tracker
— Shared Drive folder (unstructured)
— Some in Dynamics (inconsistent)
— Some in AHJ portals only
— Paper binders at regional offices
— Missing entirely for 20-40% of units

What it must become

cert_type: "Operating Certificate"
issue_date: "2024-05-01"
expiry_date: "2025-05-01"
issuing_ahj: "FL-MIA-DADE"
cert_status: "Current"
cert_document_url: [link to PDF]
days_to_expiry: [calculated]
renewal_lead_required: 30

Step 1: Have coordinator export every row from the compliance spreadsheet into a staging table. This is the fastest source — the coordinator already has it organised.
Week 1
Step 2: Run Azure Document Intelligence against the email PDF attachments and shared drive folder — auto-extract cert type, issue date, expiry date, issuing authority from certificate PDFs. Accuracy ~90% on standard AHJ certificate formats.
AI extract
Step 3: For AHJs with public portals, run a batch lookup by permit number or address to pull current certificate status directly — fills remaining gaps without outreach.
Portal lookup
Step 4: For remaining gaps — account manager requests copy of certificate from client at next contact. Most clients have it in their building files.
Outreach

Entity 5 — Technician profiles

The compliance gate runs on this data

Tech profile cleaning is P1 because the entire dispatch compliance gate — the hard block that prevents uncertified dispatch — is built on this data. One missing licence expiry date means a tech could be dispatched to a job they're not legally qualified to work.

Elevator mechanic licence — number, state, expiry P1 · manual collection

The single most important field in the tech profile · hard block depends entirely on this

Current state in Dynamics

— Mostly: NULL
— Some: "licensed FL"
— Rare: actual licence number
— Almost never: expiry date
— No: multi-state tracking
— No: AHJ-specific registration

HR/Ops collects licence numbers and expiry dates directly from every active technician — this is a one-time effort. Build a simple form (Microsoft Form or Power App) techs complete on their phone with their licence details and a photo of the licence card.

Timeline: must be complete before the compliance gate goes live. Set a firm deadline — 5 business days from request.

Manual collect

Verify
