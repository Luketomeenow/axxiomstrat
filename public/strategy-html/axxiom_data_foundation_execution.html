Detailed step-by-step execution plan for Axxiom Elevator's data foundation — six phases covering audit, schema, cleaning, governance, metrics, and pipeline

Data foundation — 6-step execution

What gets built, who does it, in what order

Six sequential steps over 6 months. Each step is a prerequisite for the next. The output of Step 6 is a self-sustaining data asset that compounds automatically — every job completed adds a data point without anyone manually entering it.

6-step execution map

Step 1  Data audit — know what you have and what's missing Weeks 1–2
Step 2  Schema design — define the master data model in Dynamics Weeks 2–3
Step 3  Data cleaning — backfill, standardise, and validate historical records Weeks 3–8
Step 4  Data governance — policy, consent, access control, quality scoring Weeks 4–6
Step 5  Safety metrics framework — define the 8 metrics Axxiom owns publicly Weeks 5–7
Step 6  Automated pipeline — every future job auto-enriches the data asset Weeks 6–12

Step 1 — Weeks 1–2

Data audit — know exactly what you have

Before any cleaning or schema design, you need a complete inventory of what data exists, where it lives, how complete it is, and what's missing. This takes 2 weeks and involves both system analysis and conversations with field techs and dispatchers.

1

Inventory all data sources

Xenoz + Ops manager · Day 1–3

Day 1–3
List every system that holds elevator-related data: Dynamics 365, spreadsheets, paper logs, legacy software, email threads, third-party inspection portals, technician notebooks

Common finds: Excel tracker for one market, paper inspection binders at a regional office, email threads with certificate attachments

Critical
For each data source, document: what data it contains, date range, format, who owns it, update frequency
Required
Interview 3–4 field techs: "What do you write down after a job that doesn't go into the system?" — this surfaces shadow data that's currently lost
Key insight
Output: Data source inventory spreadsheet with completeness rating per source (1–5)

Xenoz (automated script) · Day 3–6

Day 3–6
Run a completeness script against every asset record in Dynamics — score each record 0–100% based on required fields present

Required fields: install date, equipment type, manufacturer, serial number, site address, AHJ jurisdiction, last 5 service events, last inspection date and outcome

Critical
Segment results into 3 tiers: Green (80–100% complete), Amber (50–79%), Red (below 50% — needs major backfill)
Required
Map completeness by market and client — identify which regions or accounts have the worst data quality so cleaning effort is prioritised correctly
Planning
Output: Completeness heatmap — every asset record scored, segmented by market and completeness tier

Step 2 — Weeks 2–3

Schema design — define the master data model

Before cleaning anything, define exactly what the final data structure looks like. This is the blueprint everything else is cleaned into. Done in Dynamics Dataverse — it becomes the single source of truth for all elevator asset data.

Master schema — 5 core entities in Dataverse

Asset record Master entity

Reqasset_id · equipment_type · manufacturer · model · serial_number · install_date · site_id · ahj_jurisdiction
Reqfloor_count · drive_type (hydraulic/traction/MRL) · last_modernisation_date
Optiot_sensor_id · warranty_expiry · rated_capacity_kg · rated_speed_mps

Service event record Linked to asset

Reqevent_id · asset_id · event_type (PM/repair/inspection/emergency) · event_date · tech_id
Reqduration_mins · fault_codes[] · parts_replaced[] · outcome (pass/fail/partial)
Opttech_notes · follow_up_required · client_notified · cost_usd

Compliance record Linked to asset

Reqcompliance_id · asset_id · cert_type · issue_date · expiry_date · issuing_ahj
Reqcert_status (current/expired/pending) · inspection_result · inspector_license_num
Optcert_document_url · renewal_submitted_date · deficiency_codes[]

Site record Linked to client

Reqsite_id · client_id · address · state · county · zip · building_type · ahj_jurisdiction
Reqfloor_count · occupancy_type · access_requirements[]
Optfacility_manager_contact · bms_integration_id · annual_visit_count_target

Risk & safety score record AI-generated daily

Reqscore_id · asset_id · score_date · composite_risk_score · risk_tier (critical/watch/monitor/healthy)
Reqfactor_pm_overdue · factor_component_age · factor_fault_frequency · factor_usage_intensity
Optpredicted_failure_window_days · recommended_action · iot_anomaly_flags[]

The fault_codes[] and parts_replaced[] fields must use a standardised code taxonomy — not free-text tech notes. Define the fault code list before cleaning begins. This is what enables pattern analysis across thousands of events. "Door won't open" entered 47 different ways in free text is worthless. Fault code E14 appearing 3,200 times is signal.

Standardised taxonomy — must define before cleaning

Equipment typesHydraulic · Traction · MRL · Escalator · Platform lift
Event typesPM-routine · PM-interim · Repair · Inspection-annual · Inspection-load · Emergency · Modernisation
Fault code systemAdopt ASME A17.1 code structure + Axxiom extensions
Building typesHotel · Hospital · Office · Residential · Retail · Education · Airport · Government · Mixed-use
Outcome valuesPass · Fail · Conditional pass · Deferred · Incomplete

Step 3 — Weeks 3–8 (longest step)

Data cleaning — backfill, standardise, validate

This is the hardest and most time-consuming step. It runs for 5 weeks and is split into three parallel tracks: AI-automated cleaning, human-review cleaning, and backfill outreach. Do not underestimate the effort here — this is what makes the data publishable.

A

Track A — AI-automated cleaning (Weeks 3–5)

Xenoz builds and runs · no human review needed for these

Automated
Standardise free-text equipment types → map to taxonomy values using fuzzy match + GPT-4o

E.g. "otis gen 2 hydraulic", "Otis Gen2 Hyd", "OTIS GEN-2" all → equipment_type: "Hydraulic", manufacturer: "Otis", model: "Gen2"

AI task
Parse tech notes to extract fault codes — AI reads historical notes and maps to the ASME fault code taxonomy, confidence-scored

High-confidence matches auto-committed. Low-confidence flagged for human review queue.

AI task
Resolve site address variations → normalise to USPS standard format + append AHJ jurisdiction via address lookup API
AI task
Deduplicate asset records — identify and merge duplicate entries for the same physical unit across different market databases
AI task
Calculate derived fields: asset age from install_date, days_since_last_PM, inspection_overdue_flag — all computed automatically
AI task
Target: 85%+ of records cleaned automatically. Remaining 15% enter human review queue.
B

Track B — Human review queue (Weeks 4–7)

Ops manager + 1 data coordinator · 2hrs/day

Human review
Xenoz builds a simple review UI in Power Apps: AI presents its best guess for ambiguous records, reviewer clicks Accept / Edit / Skip

Target: 50 records/hour per reviewer. At 15% of 5,000 assets = 750 records ≈ 15 hours total review time spread over 4 weeks

Build first
Every accepted correction is fed back to improve the AI's cleaning model — human review makes the automation smarter over time
Flywheel
Output: Review queue cleared weekly. Running data quality score visible to ops manager in Dynamics dashboard.
C

Track C — Backfill outreach (Weeks 4–8)

Account managers · during normal client contact

Outreach
Priority 1 backfill: for Red-tier assets missing install date or AHJ jurisdiction, account managers ask during next scheduled visit or call — framed as "updating our records to improve your compliance tracking"
P1 gaps
Request historical inspection certificates from clients for assets missing cert records — many clients have these on file even if Axxiom doesn't
P2 gaps
For legacy assets with no service history: check manufacturer registration databases and state elevator permit records (many are public) to recover install date and initial inspection records
P2 gaps
Target: All P1 gaps resolved by Week 8. P2 gaps at 70%+ resolved. P3 gaps documented as known limitations.
D

Data quality validation — Week 8

Xenoz · automated report

Week 8
Re-run completeness scoring against the cleaned dataset. Target: 90%+ of asset records at Green tier (80%+ complete)
Gate check
Run referential integrity checks: every service event links to a valid asset, every compliance record links to a valid asset and AHJ
Gate check
Sample-audit 50 records manually: pick randomly across markets and verify that the cleaned data matches ground truth (paper records, client files)
Gate check
Gate: Data cleaning is considered complete only when the validation report shows ≥90% Green records and zero P1 gaps outstanding.

Step 4 — Weeks 4–6 (runs parallel to cleaning)

Data governance — policy, consent, access control

Governance runs in parallel with cleaning so that by the time the data is clean, the policy framework is ready. Without this, Axxiom cannot legally publish any of it or share it with regulators or insurers.

1

Write the data governance policy

Legal + Compliance officer + Xenoz · Week 4–5

Week 4–5
Define what data Axxiom can use internally (all), what can be shared with the client (their own data only), what can be published externally (anonymised aggregates only, minimum cohort size of 50 units per data point)
Core policy
Anonymisation standard: all published data is aggregated at the building-type + market level. No unit-level, client-level, or address-level data is ever published. Document this precisely so it can be shared with clients who ask
Core policy
Data retention policy: how long each record type is kept, what happens when a client contract ends, how deletion requests are handled
Required
Breach notification procedure: what Axxiom does if data is compromised — required for any client contract that includes data processing terms
Required
Output: Axxiom Data Governance Policy v1.0 — internal document, 4–6 pages, signed by CEO and Legal
2

Update client contracts with data consent clause

Legal · Week 5–6

Week 5–6
Add a standard data clause to all new contracts: Axxiom may include anonymised, aggregated service data from this account in industry safety benchmarks and reports. No client-identifiable information will be published.
All new contracts
For existing clients: send a one-page data addendum at next renewal — explain what it means in plain English, give an opt-out option. Most clients won't opt out, but having the process documented protects Axxiom legally
Existing clients
Output: Standard data consent clause + client-facing plain-English explanation document
3

Implement access control in Dynamics

Xenoz · Week 5–6

Week 5–6
Define 4 access tiers in Azure AD: Field tech (own jobs only), Account manager (their client portfolio), Ops manager (full fleet, no export), Data analyst (read-only aggregate queries, no raw PII)
Security
Tag all records with data_classification field: internal-only · client-shareable · publishable-aggregate. Any export or report query is filtered by this tag automatically
Security
Enable audit logging on all data access and exports — who queried what, when, and what they exported. Required for both internal compliance and regulator/insurance credibility
Audit trail
Output: Azure AD roles configured · data classification tags on all records · audit log active

Step 5 — Weeks 5–7 (runs parallel)

Safety metrics framework — the 8 metrics Axxiom owns

These are the specific, named, publicly-defined metrics that Axxiom will calculate, track, and publish. Defining them precisely before the pipeline is built ensures the pipeline captures exactly what's needed to calculate them.

The 8 Axxiom safety metrics — definitions and data requirements

M1  Inspection pass rate % inspections passing first-time, by vertical + market
M2  Mean time between failures (MTBF) Avg days between unplanned outages, by equipment type + age cohort
M3  PM compliance rate % of required PMs completed on schedule, by client and market
M4  Emergency call rate Emergency events per 100 units per year, by building type
M5  Top 10 fault codes by frequency Most common fault codes fleet-wide and by equipment type, updated quarterly
M6  Certificate compliance rate % of units with current operating certificate, by AHJ jurisdiction
M7  Age-risk correlation index Failure rate vs. unit age — shows at what age risk accelerates for each drive type
M8  Post-PM failure reduction rate % reduction in emergency calls in the 90 days following a PM visit — proves PM value

M8 is the most powerful metric strategically. It directly quantifies the ROI of preventive maintenance — "buildings on Axxiom's PM program had 43% fewer emergency outages in the 90 days following a PM visit." This is the data that sells contract renewals, justifies premium pricing, and gets cited by insurance carriers when pricing policies for Axxiom-serviced buildings.

→

For each metric: define → validate → automate

Xenoz + Compliance officer · 3 iterations

3 steps each
Define: Write the precise calculation formula, minimum data requirements, and exclusion criteria (e.g. M2 excludes units with fewer than 24 months of history — insufficient sample)
Week 5
Validate: Run the formula against the cleaned dataset. Do the results make sense? Are there anomalies that suggest data quality issues? Adjust formula or flag data issues.
Week 6
Automate: Build the calculation as a scheduled Power Automate flow or Azure Function that recalculates each metric on a defined cadence (daily for operational metrics, monthly for trend metrics)
Week 7
Output: Axxiom Safety Metrics Framework v1.0 — a document defining each metric precisely, suitable for sharing with regulators and insurers as methodology documentation

Step 6 — Weeks 6–12

Automated pipeline — the data flywheel that never stops

This is the step that makes the foundation self-sustaining. Every future job, inspection, fault code, and PM visit automatically enriches the data asset without anyone manually entering it. After Week 12, the data compounds on its own.

1

Field tech mobile capture — structured input at job close

Xenoz builds · Ops manager pilots with 5 techs first

Weeks 6–8
Replace the free-text "tech notes" field with a structured job close form in Dynamics mobile / Teams: fault code picker (searchable dropdown using ASME taxonomy), parts replaced (searchable parts catalogue), outcome selector, follow-up flag

The form must take under 2 minutes to complete. If it's slower than typing notes, techs won't use it.

Critical UX
Add a voice-to-text option for tech notes — tech speaks their observation, Azure Speech converts to text, AI extracts fault codes from the transcript and pre-fills the form. Tech reviews and submits.
Phase 2 option
Pilot with 5 techs for 2 weeks. Measure: completion rate, time-to-complete, data quality score of captured records vs. historical. Adjust the form UX before full rollout.
Pilot first
Output: Structured job close form live in Dynamics mobile. Every completed job = clean structured data point, zero manual data entry by office staff.
2

Inspection report ingestion — auto-parse third-party reports

Xenoz · Weeks 7–9

Weeks 7–9
Build a document ingestion pipeline: inspector submits PDF report → Azure Document Intelligence extracts structured fields (inspection date, outcome, deficiency codes, inspector license number) → auto-populates compliance record in Dynamics
High value
For AHJ portals that have API access: build a scheduled pull to sync certificate status directly — daily check on upcoming expirations
Phase 2
Output: Inspection reports are never manually entered. Every report received = compliance record updated within 1 hour of receipt.
3

Metrics dashboard — live safety intelligence in Dynamics

Xenoz · Weeks 9–11

Weeks 9–11
Build a live Power BI dashboard embedded in Dynamics showing all 8 safety metrics in real-time — filterable by market, building type, equipment type, date range
Ops visibility
Monthly automated snapshot: Power Automate saves a frozen copy of all 8 metrics on the 1st of each month. This is what gets used to generate the annual safety report — 12 monthly snapshots = one year of trend data.
Report feed
Anomaly detection: if any metric moves more than 2 standard deviations from its 90-day average, a Teams alert fires to the ops manager. This surfaces data quality issues as well as genuine safety trends.
Alerting
Final output: Self-sustaining data flywheel. Every job → structured data → metrics update → dashboard refresh. The annual safety report is now a matter of taking the 12 monthly snapshots and writing the narrative around them.
