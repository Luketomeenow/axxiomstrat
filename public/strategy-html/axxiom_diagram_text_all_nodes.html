Layer 0 — Security & Governance

Spans every layer in the architecture

All seven controls work together. They cover identity (who is allowed), secrets (where credentials live), authentication (how services prove identity), transport security (data in flight), storage security (data at rest), forensic logging (what happened), and regulatory compliance (healthcare clients).

01

Azure Entra ID

Identity & access management · the front door to everything

TitleAzure Entra ID
SubtitleSSO · RBAC · MFA · Conditional Access
What it isMicrosoft's cloud identity platform — the single authentication authority for every user, service, and AI agent in the Axxiom system.
What it doesControls who can access what across Dynamics 365, Microsoft Fabric, Power BI, and Azure OpenAI. Enforces MFA on every login. Blocks sign-ins from outside the US via Conditional Access policies. Manages 5 distinct RBAC roles — architect, data analyst, ops manager, account manager, and AI agent service principal.
Key detailAI agents authenticate as service principals — not human users. They have scoped access only: read from Dynamics, write to Fabric ax_ tables, read secrets from Key Vault. They cannot delete data.
02

Azure Key Vault

Secrets management · every API credential lives here and only here

TitleAzure Key Vault
SubtitleAll API keys & secrets stored here · 90-day rotation
What it isA FIPS 140-2 Level 2 Hardware Security Module — an encrypted vault that stores every credential the system needs: Google Ads tokens, GHL API key, CallRail key, FieldBoss Salesforce credentials, Azure OpenAI key, Qdrant key, LangSmith key.
What it doesWhen an AI agent or notebook needs a credential, it calls Key Vault at runtime via Managed Identity. No passwords are stored in code, environment files, or developer machines. Every secret access is logged — who retrieved which secret at what time.
Key detail14 secrets stored. Rotation policy: 90 days for all third-party API keys. Azure alerts 30 days before expiry for keys that require manual renewal.
03

Managed Identity

Passwordless authentication · services prove identity without credentials

TitleManaged Identity
SubtitlePasswordless auth for all agents & notebooks
What it isAzure's solution to the hardest security problem — how do you authenticate a service without giving it a password? Azure manages a short-lived certificate for each service, rotates it automatically, and uses it to authenticate with other Azure resources.
What it doesEvery Fabric notebook and LangGraph agent uses Managed Identity to access Key Vault, Dynamics 365, and Azure OpenAI. If someone stole the agent code, there is nothing in it that grants access — the identity is tied to the Azure resource, not the code.
Key detailTokens are valid for 60 minutes and issued by Azure Instance Metadata Service. No password ever exists to leak, rotate, or steal.
04

TLS 1.3 Everywhere

Transport encryption · all data in transit is unreadable to interceptors

TitleTLS 1.3 — Encryption in Transit
SubtitleAll API calls encrypted end-to-end
What it doesEvery API call — to Google Ads, GHL, CallRail, FieldBoss, and Azure OpenAI — uses TLS 1.3. Azure API Management rejects any HTTP (unencrypted) request with a redirect to HTTPS. Microsoft-to-Microsoft traffic (Dynamics → Fabric) uses Azure Private Link and never touches the public internet.
Key detailTLS 1.3 is 40% faster than TLS 1.2 due to a shorter handshake. Security and performance, not a trade-off.
05

AES-256 Encryption at Rest

Storage encryption · all data stored is unreadable without keys

TitleAES-256 — Encryption at Rest
SubtitleFabric + Dynamics + Key Vault all auto-encrypted
What it doesEvery Delta table in Fabric OneLake, every record in Dynamics Dataverse, every secret in Key Vault, and every file on the Qdrant vector store VM is encrypted using AES-256. This is automatic — no configuration required. Microsoft manages the encryption keys.
Key detailCustomer-Managed Keys (CMK) available as upgrade — Axxiom holds the encryption keys in their own Key Vault. Recommended for Phase 2 when handling hospital client data at scale.
06

Audit Trail

Three-layer forensic logging · everything that happens is recorded

TitleAudit Trail — Forensic Logging
SubtitleDynamics log · LangSmith traces · Azure Monitor
What it doesThree logging systems run simultaneously. Dynamics 365 field-level audit log records every create, update, and delete — who changed which field from what value to what, retained 3 years. LangSmith records every agent run — which prompt was sent, which tools were called, what the LLM returned — with no raw PII, only reasoning metadata. Azure Monitor captures system-level health, error rates, and Key Vault access events.
Key detailAnswers in seconds: "Who dispatched that tech to the hospital?" · "Was the licence valid that day?" · "What AI reasoning flagged that asset as critical?"
07

HIPAA-Ready

Healthcare compliance · Microsoft BAA covers all Azure services

TitleHIPAA-Ready Architecture
SubtitleMicrosoft BAA · healthcare elevator clients covered
What it doesAll six HIPAA technical safeguards are satisfied: unique user identification (Entra ID), automatic logoff (Conditional Access), encryption (AES-256 + TLS 1.3), audit controls (3-layer logging), integrity controls (append-only agent writes), and transmission security (TLS 1.3 + Private Link). Microsoft's HIPAA Business Associate Agreement covers Azure OpenAI, Fabric, Dynamics, Key Vault, and Communication Services.
Key detailCompetitive advantage: hospital and healthcare facility procurement teams increasingly require HIPAA-ready vendor infrastructure in elevator service contracts. Axxiom can say yes.

Layer 4 — AI Agents (LangGraph)

Six specialised agents orchestrated by LangGraph · each has a clear job and data requirement

LangGraph is the orchestration framework — it defines multi-agent workflows as directed graphs where each agent is a node. Graphs can be cyclical (agents loop until satisfied), conditional (different path based on data), and support human-in-the-loop checkpoints. All agents share the same LangChain foundation layer and are observed by LangSmith.

01

Dispatch & Work Order Agent

Real-time · most critical · routes techs and blocks non-compliant dispatch

TitleDispatch & Work Order Agent
SubtitleReal-time · compliance gate · geo-routing · AI job brief
TriggerNew work order created in FieldBoss → Azure Event Grid fires → agent activates in seconds.
Data it readsax_TechProfile (licence, AHJ registrations, skills, location). ax_Asset (equipment type, site address, AHJ jurisdiction). AHJ rules table (jurisdiction-specific requirements). Open work orders for scheduling conflicts.
What it doesRuns 4 compliance gates before assigning any tech: (1) Licence validity — hard block if expired. (2) AHJ jurisdiction match — hard block if tech not registered for that jurisdiction. (3) Site clearance — soft block requiring manager approval. (4) Insurance/contract — advisory flag. If all gates pass, Azure Maps calculates travel time from each eligible tech, combined with asset familiarity score, to select the optimal assignment. GPT-4o generates a structured job brief for the tech.
What it writesFieldBoss work order (tech assigned + job brief in notes). Dynamics ax_WorkOrder (status updated). Azure Communication Services SMS to assigned tech. Teams alert to dispatcher.

Below 50 (Monitor/Healthy): logged only. Score 75+ for 30 consecutive days: Modernisation Pipeline Agent triggered.

Foundation — LangChain + LangSmith

The tools every agent is built on · and the observability layer watching all of them

LangChain provides the building blocks — tools, prompts, memory, parsers, chains. Every agent is assembled from these components. LangSmith sits above all agents and records what happens at every step — not the raw data (no PII ever), but the reasoning: which prompt was sent, which tool was called, what was returned, how long it took.

01

LangChain — Tools & APIs

How agents call external systems — GHL, Dynamics, FieldBoss, Fabric

TitleLangChain Tools & APIs
SubtitleAgent connectors to GHL · Dynamics · FieldBoss · Fabric · Azure Maps
What it isLangChain Tools are the functions an agent can call during its reasoning cycle. Each tool wraps an API call — get_tech_profile() calls the Dynamics Dataverse API, create_work_order() calls the FieldBoss Salesforce API, enroll_ghl_sequence() calls the GHL REST API. The agent decides which tool to call based on the task — it is not hardcoded to call them in a fixed order.
02

LangChain Prompt Templates

Structured, versioned inputs to GPT-4o for each agent task

TitleLangChain Prompt Templates
SubtitleStructured per-agent prompts · versioned · tested in LangSmith
What it isEvery agent has specific prompt templates that define how it reasons. The Dispatch Agent's prompt instructs GPT-4o to reason as a compliance-first dispatcher. The Review Agent's prompt instructs GPT-4o to write in Axxiom's brand voice. Templates use variable substitution — the asset data, tech profiles, and client information are injected at runtime. Prompt versions are tracked in LangSmith — a prompt change is tested against historical cases before deployment.
03

LangChain Memory — Qdrant Vector Store

Agent context memory · enables personalised, history-aware responses

TitleMemory — Qdrant Vector Store
SubtitleAgent context memory · self-hosted on Azure VM · no data leaves tenant
What it doesStores vector embeddings of asset service history, client communication history, and past agent reasoning so that agents can retrieve relevant context during a task. When the Dispatch Agent handles a job for a building it has worked at 15 times, it retrieves the history of past faults at that site and includes it in the job brief. Qdrant runs on a dedicated Azure VM — self-hosted for data sovereignty.
Key detail22ms p95 query latency, 326 QPS — fast enough for real-time dispatch agent operation. AES-256 disk encryption on the Azure VM. Access restricted to agent service principal only.
04

LangChain Output Parsers

Forces structured JSON from every agent response — no free text output

TitleLangChain Output Parsers
SubtitleStructured JSON output from every agent · no free text · machine-readable
What it doesEvery agent response is validated against a defined JSON schema before being written anywhere. The Dispatch Agent must return {tech_id, assignment_reason, compliance_gates_passed, job_brief_text, estimated_arrival}. The risk scoring agent must return {asset_id, risk_score, risk_tier, top_risk_factors, recommended_action}. If GPT-4o returns malformed output, the parser catches it and retries — never writes corrupted data to Dynamics or Fabric.
05

LangSmith

Observability platform for all agents · traces, evals, costs — no PII

TitleLangSmith — Agent Observability
SubtitleTraces · evals · cost · latency · NO PII — metadata only
What it recordsFor every agent run: the exact prompt template sent (with variable names, not values), which tools were called and in what order, the tool inputs and outputs at a structural level, the final output schema, token count and estimated cost, end-to-end latency broken down by step, whether the run succeeded or failed and why.
What it never recordsRaw client data (building addresses, client names, elevator fault descriptions, tech personal details). LangSmith is configured to log only reasoning metadata — never the actual values injected into prompts. This is what keeps it safe outside Axxiom's Azure tenant.
Key capabilityPrompt versioning: when a prompt is updated, LangSmith runs the new version against a set of test cases before deployment. No prompt change goes live without passing regression testing. This is how the architect maintains quality as the system evolves.

Layer 5 — Microsoft Fabric / OneLake

The data warehouse — single source of truth · compliance-governed · Power BI reads here

Microsoft Fabric is where every data source and every AI agent write their output. It is the authoritative, encrypted, auditable store for all operational and marketing data. Power BI connects directly to the SQL analytics endpoint — no intermediate ETL, no data leakage risk. Delta format means every table has a full version history and time-travel capability.

First landing zone for all raw API data before processing

TitleStaging Lakehouse
SubtitleRaw API data lands here first · before transformation · full history preserved
What it doesThe entry point for all external data. Google Ads campaign metrics, GA4 sessions, CallRail calls, FieldBoss work orders, and GHL lead data all land here as raw Delta tables first. Raw data is preserved in its original form before any transformation — if a processing step produces an error, the raw data is still there and can be reprocessed.
