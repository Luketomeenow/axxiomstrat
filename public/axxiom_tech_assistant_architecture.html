Core decision

Knowledge Graph + Multi-Modal RAG — NOT fine-tuning as foundation

At 700GB with complex wiring schematics, cross-referenced OEM manuals, and hundreds of concurrent technicians, the architecture must be: (1) hierarchical — document → section → chunk, (2) multi-modal — text + image understanding together, (3) graph-aware — relationships between documents captured, (4) continuously improvable — feedback loop from day one.

Why NOT fine-tuning as the foundation: Fine-tuning teaches style and patterns — it does not give reliable factual recall from 700GB of documents. A fine-tuned model will confidently hallucinate a wiring procedure it partially saw during training. For safety-critical repair instructions, that is catastrophic. Fine-tuning belongs in Phase 3 only, on validated Q&A pairs, to reduce latency on common queries — not to replace document-grounded retrieval.

Layer 0 — The hardest part

700GB processing pipeline — four document types, four different pipelines

The quality of everything downstream depends entirely on how well you process the raw documents. This is where most RAG systems fail at scale. You need a different pipeline for each document type, and every image must be converted to searchable text before it can be retrieved.

1 Scanned PDFs (20% of corpus) — Azure Document Intelligence OCR pipeline
Azure Document Intelligence (formerly Form Recognizer) is the right tool here — not Tesseract. It handles complex multi-column layouts, preserves table structure, understands reading order on technical diagrams, and outputs structured JSON with bounding boxes. This matters when a wiring diagram has callouts that need to be associated with the correct component.
pipeline/scanned_pdf_processor.py
from azure.ai.documentintelligence import DocumentIntelligenceClient from azure.ai.documentintelligence.models import AnalyzeDocumentRequest from azure.core.credentials import AzureKeyCredential import json def process_scanned_pdf(pdf_path: str, doc_metadata: dict) -> dict: """ Process scanned PDF with Azure Document Intelligence. Returns structured extraction with: - Full text with preserved reading order - Tables as structured JSON - Figures with bounding boxes and captions - Page-level structure preserved """ client = DocumentIntelligenceClient( endpoint=os.getenv("AZURE_DOC_INTEL_ENDPOINT"), credential=AzureKeyCredential(os.getenv("AZURE_DOC_INTEL_KEY")) ) with open(pdf_path, "rb") as f: poller = client.begin_analyze_document( "prebuilt-layout", # preserves tables, figures, structure analyze_request=AnalyzeDocumentRequest(bytes_source=f.read()), output_content_format="markdown" # preserves structure in output ) result = poller.result() pages_data = [] for page in result.pages: # Extract figures from this page for vision processing figures = [f for f in result.figures if any(r.page_number == page.page_number for r in f.bounding_regions)] pages_data.append({ "page_number": page.page_number, "text": result.content, "tables": [t for t in result.tables if t.bounding_regions[0].page_number == page.page_number], "figures": figures, "doc_metadata": doc_metadata }) return pages_data
2 Wiring schematics + diagrams — GPT-4o Vision description pipeline Multi-modal — most critical
This is the hardest part of the entire system. A wiring schematic cannot be retrieved by text search unless it has been converted to a detailed text description. GPT-4o Vision reads each diagram and generates a structured description that captures: what type of circuit, which components, what the fault indicators mean, what the labels say, and how components connect. This description is what gets embedded and retrieved.
pipeline/diagram_vision_processor.py

3 Hierarchical chunking — 3 levels of granularity Critical for 700GB scale
At 700GB you cannot do flat chunking. You need three levels. A query about "E14 fault on KONE MonoSpace" should find the right DOCUMENT first, then the right SECTION, then the right CHUNK — not scan 2 million chunks blindly. This is the single most important scaling decision.
pipeline/hierarchical_chunker.py — three-level structure
from dataclasses import dataclass from typing import List @dataclass class DocumentChunk: # Level 0 — Document summary (one per document) doc_id: str # unique document identifier doc_summary: str # GPT-4o generated 200-word summary equipment_types: list # ["hydraulic", "traction", "MRL"] manufacturers: list # ["KONE", "Otis", "Schindler"] model_numbers: list # ["MonoSpace", "Gen2", "MicroDrive"] document_type: str # OEM_MANUAL | ASME_CODE | AHJ | AXXIOM_SOP doc_title: str @dataclass class SectionChunk: # Level 1 — Section (one per section/chapter) section_id: str doc_id: str # parent document section_title: str # "Chapter 4: Door Operator System" section_summary: str # 100-word GPT-4o summary section_path: str # "4 > 4.2 > 4.2.1" fault_codes: list # fault codes found in this section page_range: tuple # (start_page, end_page) @dataclass class ContentChunk: # Level 2 — Actual content chunk (text or diagram description) chunk_id: str section_id: str # parent section doc_id: str # grandparent document text: str # raw chunk text contextual_text: str # text with contextual prefix for embedding chunk_type: str # "text" | "table" | "diagram_description" page_number: int image_path: str # if diagram — path to original image related_chunk_ids: list # "see Figure 3-4" → links to that diagram chunk

Layer 1 — The secret ingredient

Knowledge Graph — capturing relationships pure vector search cannot

When a technician asks "elevator door won't close on Unit 47" — pure vector search returns relevant text. But the knowledge graph knows that fault code E14 is associated with door operator systems, that door operator docs reference wiring schematic W-14-D, and that the KONE MonoSpace model has a specific door motor variant. That traversal is what produces a genuinely useful answer.

Layer 2–3 — Multi-stage retrieval

Five-stage retrieval pipeline — narrows from 2M+ chunks to 5 perfect ones

The query "I'm seeing a blinking light on E14 — what does that mean?" needs to go through structured entity extraction (fault code: E14), hierarchical vector search (find the right document first), knowledge graph traversal (related procedures, diagrams, components), cross-modal retrieval (get the wiring diagram), and Voyage reranking. Not a single vector search.

1 Full retrieval pipeline — all 5 stages Core retrieval
retrieval/multi_stage_retriever.py
import anthropic, voyageai, json from qdrant_client import QdrantClient from neo4j import GraphDatabase claude = anthropic.Anthropic() voyage = voyageai.Client() qdrant = QdrantClient(url=..., api_key=...) neo4j = GraphDatabase.driver(...) def retrieve(query: str, equipment_context: dict = None) -> dict: """ 5-stage retrieval pipeline. Returns: relevant chunks + original images + confidence score. """ # ── STAGE 1: Query Understanding ────────────────────────── # Extract structured entities BEFORE searching # "blinking light on E14" → fault_codes:["E14"], symptom:"blinking light" entities = _extract_query_entities(query) # ── STAGE 2: Knowledge Graph Traversal ─────────────────── # Jump directly to fault code E14 in the graph # → get associated equipment models, sections, procedures graph_context = None if entities.get("fault_codes"): graph_context = _graph_lookup(entities["fault_codes"][0]) # ── STAGE 3: Hierarchical Vector Search ────────────────── # Level 0: Find matching DOCUMENTS first (not chunks) doc_filter = _build_qdrant_filter(entities, equipment_context) doc_hits = qdrant.search( collection_name="documents_index", # document-level summaries query_vector=voyage.embed([query], model="voyage-3", input_type="query").embeddings[0], query_filter=doc_filter, limit=5, with_payload=True ) top_doc_ids = [h.payload["doc_id"] for h in doc_hits] # Level 1: Find matching SECTIONS within those documents section_hits = qdrant.search( collection_name="sections_index", query_vector=voyage.embed([query], model="voyage-3", input_type="query").embeddings[0], query_filter=_filter_by_doc_ids(top_doc_ids), limit=10, with_payload=True ) top_section_ids = [h.payload["section_id"] for h in section_hits] # Level 2: Find precise CHUNKS within those sections # Dense search dense_hits = qdrant.search( collection_name="chunks_index", query_vector=voyage.embed([query], model="voyage-3", input_type="query").embeddings[0], query_filter=_filter_by_section_ids(top_section_ids), limit=20, with_payload=True ) # BM25 sparse search (catches exact fault codes, part numbers) sparse_hits = qdrant.search( collection_name="chunks_index", query_vector=_build_bm25_vector(query), query_filter=_filter_by_section_ids(top_section_ids), limit=20, with_payload=True ) # RRF merge fused = _reciprocal_rank_fusion(dense_hits, sparse_hits) # ── STAGE 4: Cross-Modal Retrieval ─────────────────────── # If any text chunk says "see Figure X" or "refer to diagram", # fetch those linked diagram description chunks too linked_image_chunks = [] for chunk in fused[:5]: for linked_id in chunk.payload.get("related_chunk_ids", []): linked = _fetch_chunk_by_id(linked_id) if linked and linked.payload["chunk_type"] == "diagram_description": linked_image_chunks.append(linked) # ── STAGE 5: Voyage AI Reranking ───────────────────────── all_candidates = fused[:15] + linked_image_chunks texts = [h.payload["contextual_text"] for h in all_candidates] reranked_idx = voyage.rerank(query=query, documents=texts, model="rerank-2", top_k=6).results final = [all_candidates[r.index] for r in reranked_idx] return { "chunks": final, "graph_context": graph_context, "entities": entities, "images": [c.payload["original_image_path"] for c in final if c.payload.get("original_image_path")] }

Layer 4 — The product

LangGraph Tech Assistant — intent-aware, citation-mandatory, confidence-scored

The agent is not just a RAG wrapper. It classifies what the technician is asking, uses a different retrieval strategy per intent type, generates a structured step-by-step answer with mandatory citations, scores its own confidence, and flags low-confidence answers before they reach the technician.

1 Intent types and how the agent handles each one differently Agent logic

The 5 intent types — different retrieval strategy per type

FAULT_CODE  "I'm seeing E14"Graph lookup first → jump directly to fault code node → get procedure, documents, diagrams
TROUBLESHOOT  "Door won't close"Symptom-based retrieval → semantic search on symptom language → generate step-by-step diagnostic tree
SPECIFICATION  "What oil does this take?"Model-specific filter → retrieve from manufacturer documents first → return exact spec with source
PROCEDURE  "How do I adjust the door speed?"Procedure retrieval → return numbered steps → always include safety precautions first
COMPLIANCE  "What does ASME require for this?"Filter to ASME_CODE + AHJ documents only → retrieval from regulatory sources with exact code citations
2 Answer generation — structured, cited, confidence-scored Output format
agents/tech_assistant_agent.py — answer generation
def generate_answer(state: TechAssistantState) -> TechAssistantState: context = "\n\n".join([ f"[SOURCE: {c.payload['doc_title']} | {c.payload['section_path']} | Page {c.payload['page_number']}]\n{c.payload['text']}" for c in state["retrieved_chunks"] ]) prompt = f"""You are Axxiom Elevator's AI technical assistant. A field technician needs help. Answer ONLY from the provided source documents. TECHNICIAN QUERY: {state['query']} QUERY TYPE: {state['intent']} EQUIPMENT CONTEXT: {state.get('equipment_context', 'Not specified')} SOURCE DOCUMENTS: {context} RESPONSE FORMAT — always follow this exact structure: ## ⚠️ SAFETY FIRST [Any safety precautions relevant to this task] ## Answer [Direct answer to the question in plain language] ## Step-by-Step 1. [Step with specific action] 2. [Step with specific action] ... ## Relevant Diagrams [Reference any diagram from the sources that applies] ## Sources - [Document title, Section, Page number] for each claim ## Confidence: [HIGH / MEDIUM / LOW] [One sentence explaining confidence level. LOW = conflicting information in sources, or query exceeds what documents cover] RULES: - Never answer from your own training knowledge — only from the provided sources - Every factual claim must have a source citation - If the documents do not contain the answer, say exactly that - If safety is involved, always mention it first - Use plain language a field technician understands""" response = claude.messages.create( model="claude-sonnet-4-20250514", max_tokens=2500, messages=[{"role":"user","content":prompt}] ) state["answer"] = response.content[0].text # Parse confidence for flagging pipeline confidence = "HIGH" if "Confidence: LOW" in state["answer"]: confidence = "LOW" state["flag_for_review"] = True # → SME review queue elif "Confidence: MEDIUM" in state["answer"]: confidence = "MEDIUM" state["confidence"] = confidence return state

Layers 5–6 — Getting smarter + creating Axxiom IP

Continuous learning loop + document and image generation

1 Continuous learning — feedback → validation → knowledge improvement Gets smarter

The learning pipeline — four stages

Stage 1 — Feedback captureThumbs up/down after every answer. Tech notes: "this worked" or "this missed X". Logged to Fabric rag_feedback_log.
Stage 2 — Automatic flaggingLOW confidence answers + negative feedback → SME review queue automatically. Not manually curated.
Stage 3 — SME validationExpert technician or compliance officer reviews flagged answers. Marks correct/incorrect. Adds notes. This creates labelled training data.
Stage 4 — Knowledge base updateValidated correct answers stored as new internal Axxiom knowledge chunks. These are TRUSTED above OEM manuals in future retrievals.
The compounding effect: Month 1 the system answers from OEM documents. Month 6 it has 2,000 validated Axxiom Q&A pairs. Month 12 it has 10,000+ and those validated pairs are retrieved with higher trust weight than raw OEM text. By Phase 3, you have enough labelled data to fine-tune a domain-specific model.

The correct approach is: GPT-4o Vision reads the original → Claude generates structured SVG code representing the schematic → human expert validates → SVG is stored as an Axxiom asset.

The one thing that must not be skipped

GPT-4o Vision processing every diagram before launchIf diagrams are not described in text form, they are invisible to the retrieval system. A tech asking about a wiring issue gets text-only answers with no diagram. This step takes time — budget 4–8 weeks for 700GB.
Hierarchical index — not flat chunksFlat chunking 700GB creates 2M+ chunks. Searching all of them for every query is slow and noisy. The three-level index is not optional at this scale.
Human review before any generated document is trustedEvery Axxiom-generated SOP, procedure, or schematic must be reviewed by a licensed mechanic or compliance officer before it enters the AXXIOM_VALIDATED tier. No exceptions.
